# 项目记忆

本文件用于沉淀项目开发中可复用的事实、问题结论和工程原则，帮助后续开发避免重复踩坑。

## 使用约定

1. 开始任务前先阅读本文件；完成任务后，若产生了可复用的新结论，应同步更新。
2. 只记录已确认、对后续工作有价值的内容；临时过程、聊天记录和未经验证的猜测不写入。
3. 问题记录应包含现象、原因、解决方案和预防措施，不能只写“已修复”。
4. 原则应具体、可执行、可验证；失效或被替代的内容应及时修改或删除。
5. 不记录密码、令牌、私钥、个人信息或其他敏感数据。

## 项目事实

- 当前仓库包含后端表结构文件：`山海灵感便利店-后端表结构.xlsx`。
- Excel 定义了 28 张表、236 个字段；当前数据库目标为 MySQL 8。
- 后端采用 Node.js 20+、TypeScript、Express 和 `mysql2`，接口基础路径为 `/api/v1`。
- `database/schema.sql` 是可执行的数据库结构，字段以 Excel 为准，索引、外键和默认值属于实现补充。
- App 领域接口依据 `API_IMPLEMENTATION_TASK.md`；用户会话 Token 与管理端 `API_TOKEN` 是两套独立身份体系。
- 幂等下单和退款依赖 `database/migrations/001_domain_support.sql` 中的 `idempotency_requests`。

## 工程原则

### P-001：先确认现状，再实施修改

- **原则**：修改代码、配置或数据结构前，先检查仓库现状和已有约定。
- **目的**：避免重复建设、覆盖既有实现或引入风格不一致的问题。
- **验证**：提交结果应说明检查过的相关文件，并保持改动范围与需求一致。

### P-002：问题修复必须形成闭环

- **原则**：修复问题时，同时确认根因、影响范围和验证方法；适合自动化验证的，应补充测试。
- **目的**：防止只处理表面现象，导致相同问题再次出现。
- **验证**：问题记录应完整填写“根因、解决方案、验证、预防措施”。

### P-003：数据库结构以项目约定的权威来源为准

- **原则**：涉及表、字段、类型或关系时，先核对当前表结构文件及后续确定的迁移方案，不凭印象创建。
- **目的**：减少代码模型、接口和数据库定义不一致。
- **验证**：相关实现中的表名、字段名、类型和约束可追溯到权威结构定义。

### P-004：交易流程不能用通用 CRUD 拼接

- **原则**：下单、扣库存、领券、支付回调和退款必须使用事务化领域接口，并显式实现幂等与状态流转。
- **目的**：避免部分成功导致订单、库存、优惠券和资金状态不一致。
- **验证**：领域接口应包含事务测试、重复请求测试和非法状态流转测试。

### P-005：SKU 库存只扣一个权威来源

- **原则**：订单项有 SKU 时只使用并更新 `product_skus.stock`；无 SKU 时使用 `products.stock`，禁止一次销售同时扣减两处。
- **目的**：避免商品总库存与 SKU 库存含义未定义时发生双重扣减。
- **验证**：下单和取消必须对同一库存来源做方向相反的流水；变更规则前需先明确商品总库存是否为 SKU 聚合值。

### P-006：用户 Token 与管理 Token 不可混用

- **原则**：App 接口从 `user_sessions` 验证用户并取得 `user_id`；`/admin` 和 `/resources` 仅接受 `API_TOKEN`。
- **目的**：防止普通用户利用通用 CRUD 修改库存、价格、订单状态或他人数据。
- **验证**：所有用户写接口均忽略客户端 `user_id`，未配置 `API_TOKEN` 时管理接口必须拒绝访问。
